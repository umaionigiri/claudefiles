# 大阪市統合PF 月次納品フォルダ作成スキル

毎月の納品作業を始める時、当月分のフォルダ階層と書式付き Office 文書を半自動で生成する。テンプレートは `assets/templates/` に同梱されており、シート名・日付・履行内容のみを納品月に合わせて書き換える。**書式（セル色・枠線・フォント）は破壊しない**。

## いつ使うか
- ユーザーが「5月分の納品フォルダ作って」「統合PFの納品準備して」など、月次納品作業の開始を依頼した時
- 既存の `納品ベースフォルダ` 配下に、過去の `YYYYMMDD_n回目` フォルダが並んでいる状態で、次の回の枠を作る必要がある時

## 厳守ルール（最重要）
- **既存の納品フォルダ・ファイル（`/mnt/c/Users/shoken.ohshiro/Accenture/大阪市_統合PF-XX_管理者(Mup) - 01_納品/` 配下の既存サブフォルダおよびその中身）には一切変更を加えない**。テンプレートはスキル同梱版（`assets/templates/`）のみを使用する。
- **顧客環境のフォルダには絶対に書き込まない（READ-ONLY 厳守）**。以下2つのSharePointルートが該当：
  - `/mnt/c/Users/shoken.ohshiro/osakacitycommunication/B2B_大阪市統合プラットフォーム導入・構築 - ドキュメント/` （会議資料、議事録、各種管理簿、稼働状況報告書等）
  - `/mnt/c/Users/shoken.ohshiro/osakacitycommunication/B2B_大阪市統合プラットフォーム連絡チーム - ドキュメント/` （サービス横断会議連絡台帳等）
  許可される操作は **read-only** のみ（`iterdir()`、`shutil.copytree(src,…)`、`shutil.copyfile(src,…)`）。ソース側にはタイムスタンプ含めて一切の書き込みを発生させない。新しい顧客環境を扱う場合も同様にこのルールを適用する。
- **確認は Step 0 のパラメータのみ。それ以降は一気通貫**。Step 0 でユーザーから明示的に「進む」確認を取った後は、各ステップ間で `AskUserQuestion` を挟まず `all` モードで Step 1〜9 + サマリ md 生成までを 1 コマンドで完了させる。途中ステップで都度確認を求めない。
- **サマリ md（`納品作成サマリ_<実行日YYYYMMDD>.md`）は必ず生成する**。`all` モードまたは `summary` アクションを使うこと。個別 stepN を 1 つずつ呼ぶ運用は基本禁止（テンプレート修正等のデバッグ時のみ）。デバッグで個別 step を呼んだ場合も、最後に必ず `summary` アクションを呼んでサマリを生成して終わる。

## スキルの実行手順（2 ステップだけ）

### Step 0: パラメータ計算 + ユーザー確認

ユーザーから「納品月」を聞き出す（「5月分」「2026-05」「202605」などを許容）。

```bash
python3 ~/.claude/skills/delivery-folder-creator/scripts/create_delivery_folder.py compute \
  --month "<納品月>" \
  --base "/mnt/c/Users/shoken.ohshiro/Accenture/大阪市_統合PF-XX_管理者(Mup) - 01_納品"
```

返ってくる JSON から以下を読み取り、ユーザーに表で提示：
- `n` — 今回の回数（既存フォルダから自動取得した値+1）
- `root_name` — 作成予定フォルダ名（**納品月の月末日**ベース。例: 4月分 → `20260430_第7回目`）
- `month_end_iso` — 納品月の月末日（業務完了報告の日付）
- `next_month15_iso` — 翌月15日（請求書の日付）
- `reiwa_fiscal` — 履行内容に使う令和年度

`AskUserQuestion`「この内容で当月納品フォルダ作成に進む?」 → YES なら **Step 1（all モードで一気通貫）** へ。NO ならここで停止。

### Step 1: 一気通貫実行（all モード）

Step 0 で取得した `n` と `today_str`（= `month_end_str`）を `--n` / `--today` にそのまま渡し、`all` アクションで Step 1〜9 + サマリ md 生成を 1 コマンドで完了させる。

```bash
python3 ~/.claude/skills/delivery-folder-creator/scripts/create_delivery_folder.py all \
  --month "<納品月>" \
  --base "/mnt/c/Users/shoken.ohshiro/Accenture/大阪市_統合PF-XX_管理者(Mup) - 01_納品" \
  --n <n> --today <YYYYMMDD>
```

`all` モードは内部で以下を順次実行する。**各ステップ間で確認を取らない**：

1. ルートフォルダ作成（`<月末日>_第n回目`）
2. サブフォルダ階層作成（`第n回中間分成果物/` 配下のツリー）
3. 会議資料コピー（顧客環境 → `会議資料/{作業部会, 進捗会議}/`、議事録ファイルは除外）
4. 議事録抽出（`議事録/{作業部会, 進捗会議}/`、v1.00 のみ採用、未確定は `※確認中のため次月納品.txt` プレースホルダ）
5. 管理簿コピー（リスク・課題ToDo・サービス横断会議連絡台帳の 3 ファイル）
6. 稼働状況報告書コピー（納品月の前月度フォルダ）
7. 検査仕様書 xlsx 生成（テンプレート → シート名のみ書換）
8. 業務完了報告 docx 生成（日付 + 履行内容を書換）
9. 統合PF請求書 docx 生成（請求日付を書換）
10. 納品作成サマリ md 生成（自動）

完了後、JSON 出力の最後の要素 `summary_md` のパスをユーザーに提示し、Excel/Word での目視確認を促す。

## 各ステップの詳細仕様

### Step 2 で作成されるフォルダツリー

```
第n回中間分成果物/
  ├ プロジェクト管理/
  │   ├ リスク管理表/
  │   ├ 課題管理表/
  │   ├ 会議資料/{作業部会, 進捗会議}
  │   └ 議事録/{作業部会, 進捗会議}
  ├ ガバナンス検討/   (空)
  └ 本稼働前対応/{事業者間質問管理表, 稼働状況報告書, 本稼働前対応計画書}
```

### Step 3: 会議資料コピーの命名パターン

| ソース（READ-ONLY） | 命名パターン | 配置先 |
|-------|-----------|-------|
| `01_作業部会` | `第N回作業部会_YYYYMMDD…` | `…/会議資料/作業部会/` |
| `02_進捗会議` | `第N回進捗会議_YYYYMMDD…` | `…/会議資料/進捗会議/` |

- **日付部分の年月が納品月と一致するフォルダ**を丸ごとコピー
- **議事録ファイル（ファイル名に「議事録」を含む）は除外**して会議資料側には残さない（議事録は Step 4 で別フォルダに格納）
- 該当フォルダが見つからない場合は `status: no_match` で何もしない

### Step 4: 議事録抽出のロジック

各会議フォルダの **トップレベル直下** にある「議事録」ファイルを判定し、`…/議事録/{作業部会, 進捗会議}/第N回…_YYYYMMDD/` 配下に格納：

- ファイル名に `v1.00`（1.0版）を含む議事録 → そのままコピー
- v1.00 が無い／議事録ファイル自体が無い（書面開催等） → `※確認中のため次月納品.txt` を作成（本文に理由を記載）
- `old/` 配下の旧バージョンは対象外

#### 前回納品からの議事録繰越（重要）

前回納品（n-1）時点で議事録が確定していなかった会議について、当月で v1.00 が出ていれば自動で繰越コピーする。検出は2つのケースを両方カバー：

- **Case A: 明示的なプレースホルダ**: 前回納品の `議事録/<label>/<会議>/` に `※確認中のため次月納品.txt` がある場合、その会議をお客さま環境で再判定し、v1.00 が確定していれば今回の `議事録/` に取り込む。
- **Case B: 議事録フォルダ未作成**: 前回納品の `会議資料/<label>/<会議>/` には会議フォルダがあるのに、`議事録/<label>/` 配下にその会議フォルダ自体が存在しないケース（旧版スクリプトでプレースホルダ未配置のまま納品されたパターン）。この場合も今回の `議事録/` に取り込む。

両ケースとも、お客さま環境のソースが消えていれば `carried_over_source_missing`、まだ未確定なら `carried_over_still_pending` として状態を記録する（前回納品側のプレースホルダはそのまま）。

### Step 5: 管理簿コピー（3 ファイル）

| ソース | 配置先 | 出力ファイル名 |
|-------|-------|--------------|
| `…/B2B_導入・構築/General/00_共通/00_プロジェクト管理/30_リスク管理/【統合PF】全体_リスク管理簿.xlsx` | `…/プロジェクト管理/リスク管理表/` | `【統合PF】全体_リスク管理簿_<月末YYYYMMDD>.xlsx` |
| `…/B2B_導入・構築/General/00_共通/00_プロジェクト管理/40_課題・ToDo管理/【統合PF】全体_課題・ToDo管理簿.xlsx` | `…/プロジェクト管理/課題管理表/` | `【統合PF】全体_課題・ToDo管理簿_<月末YYYYMMDD>.xlsx` |
| `…/B2B_連絡チーム/03_統合PFサービス横断会議連絡/00_マスタ/【共同編集】【統合PF】サービス横断会議_連絡台帳.xlsx` | `…/本稼働前対応/事業者間質問管理表/` | `【共同編集】【統合PF】サービス横断会議_連絡台帳_<月末YYYYMMDD>.xlsx` |

ピュアなファイルコピー（書式・内容を一切変更しない）。3エントリ目は `B2B_連絡チーム` 環境（READ-ONLY）から取得する。

### Step 6: 稼働状況報告書コピー

顧客環境 `…/General/02_ガバナンス検討/09_稼働状況報告書/<YYYY年M月度>/` のうち、**納品月の前月度**フォルダをフォルダ丸ごと `…/本稼働前対応/稼働状況報告書/` 配下にコピー。

| 納品月 | 対象フォルダ |
|-------|------------|
| 2026年4月 | `2026年3月度` |
| 2026年5月 | `2026年4月度` |
| 2027年1月 | `2026年12月度`（年跨ぎ対応） |

### Step 7-9: Office 文書生成

| ステップ | 出力 | 書換内容 |
|--------|------|--------|
| 7 | `<月末日>_検査仕様書（様式）.xlsx`（**全角括弧**） | メインシート名を `第n回中間分` に変更（**それ以外は触らない**＝セル値・色・枠線・フォント・印刷設定すべて保持。zip+XML 直接編集） |
| 8 | `業務完了報告.docx` | P1 右寄せ日付段落: `令和X年M月D日　`（X=月末日の令和年）／ Table 0 R5C1 履行内容: `令和X年度　第n回中間分業務`（X=会計年度の令和、4-3月） |
| 9 | `統合PF請求書(第n回中間分業務).docx`（**半角括弧**、既存命名に合わせる） | Table 0 R2 セル（マージ済み右上日付）を `　令和X年　M月　15日`（翌月15日、和暦） |

## 個別 step アクション（デバッグ用途のみ）

スクリプトは `compute / step1〜9 / summary / all` の各アクションを CLI で受け付ける。通常運用は `all` 一択。個別 stepN は以下のような限定された場合のみ使用：

- スクリプト改修中の単体テスト
- テンプレート差し替え後の特定ステップだけの再実行確認

個別 step を実行した場合は、**最後に必ず `summary` アクションを呼ぶ**：

```bash
python3 ~/.claude/skills/delivery-folder-creator/scripts/create_delivery_folder.py summary \
  --month "<納品月>" --base "<base>" --n <n> --today <YYYYMMDD>
```

`summary` アクションは step1〜9 を冪等再実行（既に存在するファイル/フォルダは上書きしない）した上でサマリ md を書き出す。すでに納品物が揃っていれば副作用ゼロで md だけが生成される。

## サマリ md の中身

`<root>/納品作成サマリ_<実行日YYYYMMDD>.md` に以下を含む：

- 実行情報（納品月・回数・月末日・翌月15日・令和年度）
- 作成パスツリー
- 9 ステップそれぞれの結果（コピー件数・配置先・置換内容・プレースホルダ・前回繰越）
- 「次に対応してください」セクション（Office 文書の目視確認・議事録未確定項目・管理簿の最新性確認）

ユーザーは納品作業の最後にこの md を読めば、何が自動で行われ、何を人手で確認すべきかが一目で分かる。

## 依存

- Python 3.12+
- `python3-openpyxl`、`python3-docx`（apt パッケージ。venv 不使用）

## トラブルシューティング

- **n が想定と違う**: ベースフォルダに過去の回フォルダが揃っているか確認。検出ロジックは `^\d{8}_(?:第)?(\d+)回目$`（「第」あり/なし両対応、「リセール」等の特殊フォルダはスキップ）。手動で n を指定したい場合は `--n` を渡す。
- **書式が崩れた**: テンプレート (`assets/templates/`) を再度ユーザー手元のオリジナルから cp し直す。スクリプトは「シート名」「該当 run のテキスト」しか触らないため、テンプレを差し替えれば書式問題は再現しない。
- **同じ実行日に2回目を作成したい**: ルートフォルダ名は月末日 + n。`all` モードは冪等（既存フォルダは status=exists 扱い）なので再実行で破壊しない。再生成したい場合は `--n` を手動指定するか、既存フォルダを別名にリネームしてから再実行。
- **サマリ md が出来ていない**: `all` モードを使ったか確認。個別 stepN だけを実行した場合は `summary` アクションを最後に呼ぶ。
- **前回納品の議事録繰越が漏れた**: 前回納品の `議事録/<label>/` を確認。Case B（議事録フォルダ自体が無い）に該当しないか目視で確認。スクリプトは Case A・Case B 両方を検出するが、お客さま環境にソース会議フォルダがすでに無い場合は手動補填が必要。
- **ライブラリ未インストール**: `sudo apt install python3-openpyxl python3-docx` を実行してもらう。
