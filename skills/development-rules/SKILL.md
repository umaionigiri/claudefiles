# Development Rules

コード品質を担保するための開発ルールスキル。

## ワークフロー

```mermaid
flowchart TD
    A[タスク受領] --> B[Context7 リサーチ]
    B --> C[設計検討]
    C --> D{複雑度判定}
    D -->|シンプル| E[直接実装]
    D -->|複雑| F[分割設計]
    F --> G[並列タスク分析]
    G --> H[独立タスク並列実行]
    H --> I[依存タスク順次実行]
    E --> J[コード実装]
    I --> J
    J --> K[原則チェック]
    K --> L{原則違反?}
    L -->|Yes| M[リファクタリング]
    M --> K
    L -->|No| N[完了]
```

## Step 1: Context7 リサーチ（必須）

**実装前に必ず実行:**

```
mcp__context7__resolve-library-id({ libraryName: "framework-name" })
mcp__context7__query-docs({ libraryId: "/lib/xxx", topic: "implementation pattern" })
```

- 最新のAPI・パターンを確認
- 非推奨機能の回避
- ベストプラクティスの把握

## Step 2: 設計検討

- 要件を明確化
- 既存コードとの整合性確認
- 影響範囲の特定

## Step 3: 並列タスク分析

| 判定 | 処理方法 |
|------|----------|
| 依存関係なし | Task tool で並列実行 |
| 依存関係あり | 順次実行 |

## Step 4: コード実装

原則に基づいた実装を行う。

## Step 5: 原則チェック

実装後、以下の原則に違反していないか確認。

## クイックリファレンス

### SOLID 原則早見表

| 原則 | 略称 | 内容 | 違反例 |
|------|------|------|--------|
| Single Responsibility | SRP | 1クラス/関数 = 1責務 | 複数処理を1関数に詰め込む |
| Open-Closed | OCP | 拡張に開、修正に閉 | 条件分岐の追加で既存コード変更 |
| Liskov Substitution | LSP | 派生クラスは基底と置換可能 | 継承でメソッドの契約を破る |
| Interface Segregation | ISP | インターフェースは小さく | 巨大なインターフェース |
| Dependency Inversion | DIP | 抽象に依存、具象に依存しない | 具体クラスを直接 new |

### その他の原則

| 原則 | 内容 | 実践 |
|------|------|------|
| YAGNI | 必要になるまで実装しない | 「念のため」機能を作らない |
| DRY | 知識の重複を避ける | 3回繰り返したら抽象化検討 |
| KISS | シンプルな解を選ぶ | 複雑なコードはリファクタ対象 |

### コード品質基準

| 項目 | 基準 |
|------|------|
| 命名 | 意味のある名前 |
| エラー処理 | 適切なハンドリング |
| コメント | 「なぜ」を説明（「何」ではない） |
| マジックナンバー | 定数化 |
| 補足説明 | 日本語コメント可 |

## 検証チェックリスト

### 実装前チェック

- [ ] Context7 でリサーチ済み
- [ ] 既存コードのパターンを確認済み
- [ ] 影響範囲を特定済み

### 実装後チェック

- [ ] SRP: 各関数/クラスは単一責務か
- [ ] OCP: 拡張可能な設計か
- [ ] YAGNI: 不要な機能を追加していないか
- [ ] DRY: 重複コードはないか
- [ ] KISS: 最もシンプルな解か
- [ ] 意味のある命名か
- [ ] 適切なエラー処理があるか

### コードレビューチェック

- [ ] ロジックの正確性
- [ ] エッジケースの考慮
- [ ] セキュリティリスクの有無
- [ ] パフォーマンスへの影響
- [ ] テストの有無・カバレッジ

## 重要事項

- **リサーチ必須**: 実装前に Context7 で最新情報を確認
- **過剰設計を避ける**: 依頼された内容のみを実装
- **既存パターンに従う**: プロジェクトのコードスタイルを尊重
- **コメントは日本語可**: 補足説明には日本語コメントを使用
