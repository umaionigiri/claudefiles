# PR作成スキル

以下の手順でPRを作成してください。

## 手順

### 1. 情報収集

以下を並列で実行して現在の状態を把握する:

- `git branch --show-current` でカレントブランチを確認
- `git log <base>..HEAD --oneline` で含まれるコミット一覧を取得（baseブランチは $ARGUMENTS があればそれを使用、なければリポジトリのデフォルトブランチ）
- `git diff <base>...HEAD --stat` で変更ファイルの統計を取得
- `gh pr list --head <current-branch> --state open` で既存PRがないか確認

### 2. PR本文の作成

以下のテンプレートに沿ってPR本文を作成する。各セクションはコミット内容・差分から読み取れる情報で埋める。

```markdown
## 概要
<!-- 1-3行で What（何を）と Why（なぜ）を簡潔に -->

## 関連タスク
<!-- 関連する Issue は #番号 でリンク。なければ「なし」 -->

## やったこと
<!-- 主要な変更をカテゴリごとにグルーピング。詳細リストは <details> で折りたたむ -->

## やらないこと
<!-- このPRのスコープ外を明示。レビュアーの「なぜこれは変更しなかった？」を先回り -->

## 影響範囲
<!-- 影響を及ぼす範囲。ソースコード / CI/CD / DB / 外部サービス / 他ブランチ 等 -->

## テスト
<!-- チェックボックスリストで。CI結果 + 手動確認項目 -->

## 備考
<!-- レビュアーへの伝達事項、マージ後の注意点、既知の制約 等。なければ省略可 -->
```

### 3. テンプレート記入のガイドライン

- **概要**: 結論ファーストで What + Why を1-3行。背景が複雑な場合は2段落まで可
- **やったこと**: 変更が多い場合はカテゴリでグルーピングし、ファイル単位の詳細は `<details><summary>` で折りたたむ
- **やらないこと**: スコープを明確にすることでレビュー範囲を限定する。最低1項目は書く
- **影響範囲**: 「影響なし」も明記する（レビュアーの安心材料になる）
- **テスト**: CI自動チェック項目は `[x]`、手動確認が必要な項目は `[ ]` で区別
- **備考**: マージ後に他メンバーが困りそうな注意点があれば記載。なければセクションごと省略

### 4. PR作成の実行

- `gh pr create` で作成する
- タイトルは70文字以内、プレフィックス付き（feat:, fix:, refactor:, docs:, chore: 等）
- `--base` オプションはCLAUDE.mdで明示的に指示がない限り付けない

### 5. 結果報告

作成したPRのURLを表示する。
